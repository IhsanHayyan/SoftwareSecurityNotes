CHAPTER 1

Overview of Software Security

What Is Software Security?

  • Engineering software to still work correctly even when under attack
  • Goal: Address vulnerabilities EARLY — not after building the software
  • Key quote: "Security is risk management"
  • Root causes: lack of education, poor design, bad testing, weak dev environments

Key Terminology

TermMeaning
DefectSoftware doesn't match requirements (e.g., asked for 8-char min, app allows 5)
BugCoding mistake causing crash/misbehaviour
FlawWeakness in design/logic — harder to spot than a bug
FailureSoftware can't do its job at all
VulnerabilityA weakness an attacker can actually exploit
Risk = Probability of Attack × Impact of Damage
★ Exam Topic

CIA Triad

C
Confidentiality

Only authorized people can see data. Example: Encrypting credit card numbers in the DB

I
Integrity

Data cannot be secretly changed. Example: Checksums detect file tampering

A
Availability

System/data is always accessible. Example: Preventing DoS attacks

Extended — CI4AM: Confidentiality · Integrity · Availability · Authentication · Authorization · Auditing · Management

★ Exam Topic

Quality Code vs. Secure Code

🎯 Quality Code

  • Meets user requirements
  • Performs tasks efficiently
  • Well-documented & maintainable
  • Bug-free and fast

🔒 Secure Code

  • Protects against threats & attacks
  • Ensures CIA of data
  • Input validation, safe auth
  • Resilient against SQLi, BO, etc.

Key Insight: High quality ≠ Secure. A bug-free app can still be hacked. Best practice = Quality + Security together.

CHAPTER 3

Handling Input

★ Exam Topic

Input Sanitization — Syntactic & Semantic Validation

Syntactic Validation (FORMAT)

  • Checks input looks like what you EXPECT
  • Right format, characters, length
  • Email: must have @, valid TLD, ≤254 chars
Dangerous chars to block: ` ' " \0

Semantic Validation (MEANING)

  • Checks input makes LOGICAL SENSE
  • Age=5 is syntactically valid but semantically wrong if app requires 18+
  • Email format ✓ — but does user OWN it? → Send verification link

Golden Rule: Any input from OUTSIDE your system = potentially hostile until proven otherwise. Zero trust.

★ Exam Topic

SQL Injection (SQLi)

Attacker types special SQL into input boxes → app accidentally runs it as a real DB command.

-- Attack: user types:  John' OR '1'='1
SELECT * FROM users WHERE name = 'John' OR '1'='1';
-- '1'='1' is ALWAYS TRUE → Returns ALL rows!

-- FIX — Parameterized query (Python):
cursor.execute('SELECT * FROM users WHERE name = ?', (user_input,))
-- Input treated as TEXT ONLY, never as code
MethodHow It Stops SQLi
Parameterized QueriesInput treated as TEXT, never as code
Prepared StatementsQuery structure locked first; input added separately
Hide DB ErrorsNever show raw error messages to users
WAFFilters malicious requests BEFORE they reach the DB
Keep Software UpdatedPatch PHP, Python, MySQL to fix known vulns
CHAPTER 4

Static Analysis

★ Exam Topic

Static Analysis & Static Code Analysis

Analyzing software WITHOUT EXECUTING IT — reading code to find problems before deployment.

Data Flow Analysis

Traces how data moves through the program. Best for finding INJECTION attacks — tracks where untrusted input goes and whether it's validated.

Control Flow Analysis (CFA)

Maps ALL possible execution paths. Best for unreachable code, deadlocks, and logic errors.

Abstract Syntax Trees (AST)

Represents code as a tree showing its syntax. Best for syntax errors and enforcing coding standards.

Key Tools

  • SonarQube — open source dashboard
  • Coverity — aerospace/defence level
  • Checkmarx — 20+ languages
  • Semgrep — highly customizable

Limitation: False Positives (flags safe code as bad) + False Negatives (misses real vulns). Static analysis alone is NOT enough — combine with dynamic testing and pen testing.

CHAPTER 5

Buffer Overflow

★ Exam Topic

Buffer Overflow

Analogy: A glass holds 20ml. You pour 50ml. Extra water spills onto what's around it. Buffer Overflow = more data than the buffer can hold spills into adjacent memory and corrupts it.

Dangerous vs. Safe Code (C)
// DANGEROUS — no size limit
char name[20];
scanf("%s", name);    // reads forever → OVERFLOW RISK

// SAFE — with size limit
scanf("%19s", name);   // max 19 chars + null terminator

// SAFE — strncpy with boundary check
strncpy(buf, userInput, sizeof(buf) - 1);
buf[sizeof(buf) - 1] = '\0';
ProtectionHow It Works
Stack CanariesGuard values next to return addresses. Overflow reaches canary → program aborts
ASLRRandomizes memory locations → attacker can't predict where to aim
DEP / NX BitStack/heap marked non-executable → CPU refuses to run injected code
Use Python/safer languagesAuto-manages memory → far fewer BO opportunities
CHAPTER 6

Broken Access Control & File Inclusion

★ Exam Topic

Broken Access Control

Vertical BAC

Controls based on ROLE. Attack: Regular user navigates directly to admin URL. Fix: Check role on every request.

Horizontal BAC (IDOR)

Controls based on user attributes at SAME level. Attack: Change userID=1234 → userID=5678 to see someone else's data.

BOLA

IDOR via APIs. Object-level auth checks must happen in EVERY function that uses user input to access data.

MFLAC

Missing Function Level Access Control. Hidden backend function — attacker finds and calls it. If server doesn't check auth, any user can call it.

Remediation principles: Deny access by default · Single access control mechanism · Log all access · Least privilege · Never rely on obscurity

★ Exam Topic

Local File Inclusion (LFI)

Attacker uses ../ to navigate up the folder structure and read private server files.

-- Attack URL (from demo lab):
http://localhost/vuln_demo/index.php?page=secret/admin_panel.php

// NOT SAFE — includes whatever user types
$page = $_GET['page'] ?? 'home.php';
include($page);    // DANGEROUS

// SAFE — whitelist only
$allowed = ['home', 'profile'];
if (in_array($page, $allowed)) {
    include($page . '.php');  // .php added by SERVER — safe
} else { echo "Access Denied"; }

LFI vs RFI: LFI = attacker reads local server files · RFI = server loads and EXECUTES hacker's remote file. Both fixed by whitelisting allowed page names.

XSS — Cross-Site Scripting

// Attack payload: <script>alert("XSS Attack!")</script>

// NOT SAFE — echoes raw user input
echo "Hello " . $_GET['name'];

// SAFE — htmlspecialchars() converts special chars to HTML entities
echo "Hello " . htmlspecialchars($_GET['name']);
// < → &lt;  > → &gt;  " → &quot;  Browser sees TEXT not CODE
CHAPTER 7

Error & Exception Handling

★ Exam Topic

Return Codes & Exception Handling

Return Codes

  • 0 → Success, -1 → Failure, NULL → Invalid
  • Always CHECK before using results
  • Never expose sensitive details in codes returned to users
  • Standardize: use EXIT_SUCCESS, EXIT_FAILURE constants

Exception Types

  • Checked — compile-time, e.g. IOException
  • Unchecked — runtime, e.g. NullPointerException
  • System/Fatal — OutOfMemoryError → DoS risk
  • App-Defined — custom business exceptions
Safe Exception Handling (Python)
try:
    with open('data.txt', 'r') as file:
        data = file.read()
except FileNotFoundError:
    print('File not found.')   # user-friendly — no system details
except Exception as e:
    log(e)                      # detailed log for ADMIN only
    print('An error occurred.') # generic message for USER
finally:
    print('Done.')             # always runs — cleanup

Golden Rule: Never expose internal error details to users. Log securely for administrators only.

CHAPTER 8

Broken Authentication

Authentication (AuthN)

WHO ARE YOU? — Verifying identity. Username + password, biometrics, OTP.

Authorization (AuthZ)

WHAT ARE YOU ALLOWED TO DO? — Permissions. Admin can delete; user cannot.

AttackHow It WorksDefense
Password SprayingTry COMMON passwords against MANY accountsComplexity requirements, lockout
Credential StuffingUse stolen password lists from other breachesMFA, breached password protection
Session HijackingSteal session cookie via XSS → impersonate userHttpOnly cookies, HTTPS, new session ID on login
Session ID in URLSession ID in URL leaked via browser history/logsNever put session IDs in URLs
PhishingFake trusted entity → harvest credentialsUser awareness + MFA

9 Fixes for Broken Authentication

1. Hash with bcrypt/Argon2 — never plain text

2. Enforce strong passwords (min 8 chars, complexity)

3. Strict password recovery — multi-step verification

4. Breached password protection

5. Auto-expire sessions after inactivity

6. New session ID after every successful login

7. Never put session IDs in URLs

8. MFA — require a second factor

9. User awareness training

★ Exam Topic

Deserialization Vulnerabilities

Analogy: Receiving a package that triggers a trap when opened. Attacker sends crafted data; when the app deserializes (opens) it, malicious code executes.

  • OWASP A08 — Software & Data Integrity Failures
  • App receives specially crafted data → if not checked properly, attacker executes malicious code
  • Also includes supply chain attacks — code from untrusted sources

Prevention: Validate + integrity-check all serialized data before deserializing · Use allowlists for deserializable types · Only install updates from trusted, verified sources
